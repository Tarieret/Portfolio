Human-in-the-Loop Data Analyst

A plain-English question becomes a reviewable SQL query, and nothing executes until a human approves it.

PythonDuckDBFastMCPOpenAI Responses APIStreamlit
Overview

Text-to-SQL agents that execute automatically are a liability: a misread question or an ambiguous schema can produce a query that silently returns the wrong answer, or one with unintended side effects. This project asks whether a natural-language question can become safe, reviewable SQL with a mandatory human approval step before anything runs, using the approval mechanism as a protocol-level guarantee rather than an application-level convention.

The system exposes a DuckDB database (Instacart's public order data) as an MCP server with three tools: list_tables, describe_table, and run_query. A Streamlit chat interface sends questions to OpenAI's Responses API, which calls those tools directly. Every run_query call pauses for human approval before it executes, the SQL is shown in full, and only a click resumes execution.

Architecture
Question (Streamlit chat) → OpenAI Responses API ↓ MCP tool call proposed (list_tables / describe_table / run_query) ↓ run_query only → mcp_approval_request → human reviews SQL ↓ Approve → DuckDB MCP server executes → result returned Reject → model proposes an alternative query (re-gated) ↓ Plain-English answer rendered in chat
Key Decisions
  • Chose MCP's native require_approval mechanism over hand-rolling the approval step in application code. The gate is enforced by OpenAI's infrastructure, not by this codebase, so a bug in the Streamlit script can't silently bypass it.
  • Added a second, independent safety layer inside the MCP server itself: run_query rejects anything that isn't a bare SELECT, so an approved-by-mistake destructive query still can't execute.
  • Scoped approval to run_query only, not list_tables or describe_table. Those two are read-only schema introspection and can't return anything beyond table and column names, gating them added approval clicks without adding safety. This was a revision after the first working version gated every tool call, which was safer in principle but slow enough in practice that it worked against the demo rather than for it.
  • Considered a LangChain SQL agent instead of MCP, rejected partly to keep this project's toolset distinct from the RAG project elsewhere in this portfolio (also LangChain/OpenAI), and partly because LangChain's SQL agent has no first-class human-approval primitive either, so it wouldn't have solved the actual problem.

The approval-scoping revision was the real lesson here. Blanket safety measures that gate everything equally feel rigorous, but they can be a worse design than a narrower gate placed exactly where the risk actually lives. Knowing which calls can hurt something and which can't is part of the safety design, not separate from it.

What Broke

Confirmed against a live OpenAI key and a tunneled MCP server: the approval chain worked correctly on the first real end-to-end run. Multi-step tool sequences each triggered their own independent approval request, confirming the gate is per tool call, not per conversation. Rejecting a proposed query doesn't stop the agent, it proposes an alternative, and that regenerated query still requires its own separate approval, verified by deliberately rejecting once and confirming the retry was also gated.

One infrastructure constraint shaped the deployment path: the Responses API only calls remote, HTTP-reachable MCP servers, not local stdio processes, so local development required tunneling the DuckDB server publicly rather than pointing at localhost directly.
